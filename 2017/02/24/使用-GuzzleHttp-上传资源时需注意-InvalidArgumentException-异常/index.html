使用 GuzzleHttp 上传资源时需注意 InvalidArgumentException 异常

发表于 2017-02-24

下文的 GuzzleHttp 如果没有特指版本，说的都是 6.2.2 版本

产生问题的原因

GuzzleHttp 会在上传完毕后将上传的资源进行关闭(通过 \GuzzleHttp\Psr7\Stream::__destruct())，如果是需要多次使用的资源，应该注意传递给 GuzzleHttp 的上传资源不要是 resource 或者 \Psr\Http\Message\StreamInterface 类型(实现该接口的类，实例化后返回的资源还是 resource 类型)。

解决方法

通过官方文档的 Request options 章节中 multipart 参数的说明可知，该参数只接收 \Psr\Http\Message\StreamInterface 和 resource 还有 string 类型的参数，前2种类型传递的都是 引用 ，当上传完毕 \GuzzleHttp\Psr7\Stream::__destruct() 会自动将资源销毁(通过fclose关闭资源句柄)。

传值而非引用

string 类型传递的是值而不是引用，所以销毁的只是 \GuzzleHttp\Psr7\stream_for() 函数创建的 resource 句柄。

通过传值虽然可以解决问题，但存在内存占用翻倍和每次都得重新将 string 类型转换为 \Psr\Http\Message\StreamInterface 类型的时间成本问题。

使用 Promise

\GuzzleHttp\Psr7\Stream::__destruct() 只要资源不再被使用就将其销毁，那就通过 Promise 让所有需要访问到该资源的操作都再同一个任务队列里。

更多关于 Promise 的使用方法可查看官方文档的 Concurrent requests 章节

手动销毁资源

如果不想或者业务逻辑不适合用 Promise，那也可以自己实现 \Psr\Http\Message\StreamInterface 接口来实现手动销毁资源，我们直接继承 \GuzzleHttp\Psr7\Stream 类然后重写它的 __destruct 魔术方法。
